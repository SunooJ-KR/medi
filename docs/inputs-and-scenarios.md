# 입력 체계 분석 및 사용 시나리오 기획

작성일: 2026-07-03
상위 문서: `docs/plan.md` (v2) — 본 문서는 plan.md의 입력·UX 레이어를 상세화하며, 여기서 확정된 사항은 plan.md §9 의사결정 이력에 반영됨.

## 1. 입력 표면 전체 지도

시스템에 들어오는 모든 입력을 한 눈에 정리한다. "누가·언제·어떻게"가 다른 입력을 같은 방식으로 받으면 사용성이 무너지므로, 이 지도가 스킬 UX 설계의 출발점이다.

| # | 입력 | 누가 | 빈도 | 입력 방식 | 저장 위치 |
|---|---|---|---|---|---|
| I1 | 회사 프로필 (시장·채널·경쟁사·공식몰 URL) | 사용자 | 최초 1회 + 드물게 수정 | `/setup` 대화형 | `data/config.yaml` |
| I2 | 피부고민 택소노미 | `/setup`이 초안 생성, 사용자 확인 | 최초 1회 | 공식몰 카테고리 기반 자동 초안 → 확인 | `data/taxonomy.yaml` |
| I3 | 페르소나 | `/persona-build`가 초안, 사용자 검수 | 최초 1회 + 분기 리프레시 | LLM 초안 → 항목별 승인 | `data/personas/` |
| I4 | 내부 이벤트 | 마케터 수동 | 수시 | `/event-add` 자연어 → 구조화 확인 | `data/events/internal/` |
| I5 | 외부 이벤트 (자동) | `/event-scan` | 주간 (또는 `/brief` 내) | 웹 조사 → unreviewed 저장 | `data/events/external/` |
| I6 | 외부 이벤트 (수동) | 사용자가 직접 인지한 뉴스 | 수시 | `/event-add --external` (출처 URL 필수) | `data/events/external/` |
| I7 | 검토 큐 승인/거부 | 사용자 | 스캔 직후 | 큐 테이블에서 번호 선택 | 이벤트 `status` 갱신 |
| I8 | 시뮬레이션 파라미터 | 사용자 | 실행마다 | 플래그: `--since`, `--event`, `--backtest` | `data/reactions/<run>/` |
| I9 | 보고서 파라미터 | 사용자 | 실행마다 | 기본값: 최신 run + 직전 보고서 비교 | `reports/` |

> 분석 중 확정된 신규 요소: **I1(config.yaml)** — 기존 계획에 없었으나 event-scan이 조사 범위(경쟁사·시장·채널)를 정하려면 필수. **I6(외부 이벤트 수동 입력)** — 기존 계획은 외부=자동만이었으나, 사용자가 먼저 인지한 뉴스를 즉시 넣는 경로가 없으면 긴급 대응 시나리오(S3)가 불가능.

## 2. 입력 설계 고려 요소 (분석)

### 2.1 입력 부담 vs 데이터 품질 — 핵심 트레이드오프

마케터는 YAML을 쓰지 않는다. 그러나 자유 텍스트를 그대로 저장하면 반응 엔진의 일관성(plan.md E5)이 무너진다. 결론: **"자연어 입력 → LLM 구조화 → 사용자 확인 → validate → 저장"** 4단 파이프를 모든 수동 입력(I1, I4, I6)에 공통 적용한다.

- 사용자가 치는 것: `"다음 주 화요일부터 레티날 세럼 20% 할인 + 올리브영 단독 기획전 2주"`
- 스킬이 만드는 것: type=promotion, date, channels=[olive_young], affected_concerns(제품→택소노미 매핑으로 추론), summary
- 저장 전 반드시 구조화 결과를 표로 보여주고 확인받는다. **LLM 매핑 오류는 확인 단계에서만 잡을 수 있다** — 이 단계는 생략 불가.

### 2.2 필수 필드 최소화, 추론 가능한 것은 추론

사용자에게 직접 묻는 필수 정보는 5개로 제한: **무엇을(type), 언제(date), 어디서(market/channel), 어떤 제품/고민, 한 줄 설명.** 나머지는 추론 후 확인으로 채운다.

| 필드 | 채우는 방법 |
|---|---|
| `affected_concerns` | 제품명 → taxonomy의 제품–고민 매핑에서 추론 |
| `affected_channels` | 언급 없으면 config.yaml의 기본 채널 |
| `market` | 언급 없으면 config의 기본 시장(KR) |
| `id` | type+날짜+슬러그로 자동 생성 |

### 2.3 시간 모델 — 이벤트는 영원히 유효하지 않다

기존 계획의 누락: 3개월 전 프로모션이 이번 주 반응 추론에 계속 끼면 안 된다.

- `date_occurred` / `date_collected` 분리(기존 E6)에 더해, **simulate는 기본 lookback window(8주) 안의 이벤트만 입력**으로 삼는다. `--since 2026-05-01`로 조정 가능.
- 규제 변화처럼 장기 유효한 이벤트를 위해 이벤트에 선택 필드 `active_until`을 두고, 있으면 lookback 대신 이 값을 따른다.

### 2.4 내부 이벤트의 `planned` 상태 (확정: 허용)

내부 이벤트 `status: planned | occurred`.

- **용도**: 실행 전 사전 점검(시나리오 S2). 프로모션·출시 계획을 미리 넣고 페르소나 반응을 본 뒤 계획을 수정한다. 이것이 도구를 "사후 분석기"가 아닌 "의사결정 도구"로 만든다.
- **오염 방지 장치**: ① 보고서에서 planned는 "사전 점검" 배지로 명확히 구분 렌더 ② 변화 추적(diff) 통계와 백테스트에서 제외 ③ planned 이벤트가 실행되면 `/event-add --confirm <id>`로 occurred 전환(이때 날짜·내용 수정 기회 제공).
- 데모의 백테스트-단독 원칙(plan.md §9-6)과 충돌하지 않음: 그 결정은 데모 시나리오에서 가상 외부 이벤트를 배제한 것.

### 2.5 중복·수정·삭제

- **중복 감지**: 저장 전 같은 type + 날짜 ±7일 + 제목 유사 이벤트가 있으면 경고하고 "새로 저장 / 기존 수정 / 취소"를 묻는다. `/event-scan`도 기존 이벤트와 동일 출처 URL이면 스킵.
- **수정**: `/event-add --edit <id>` 로 동일한 구조화-확인 파이프를 재사용. 파일 직접 수정도 가능(validate가 커밋 전 검사).
- **삭제 대신 상태 전환**: 외부 이벤트는 `rejected`, 내부 이벤트는 파일 삭제 허용(git이 이력 보존).

### 2.6 언어와 enum

입력·확인 대화는 한국어, 저장되는 enum·id는 영문. 매핑은 스킬(LLM)이 담당하고 확인 표에는 둘 다 표기한다(예: `promotion (프로모션)`). 이유: 파일은 코드가 읽고(정렬·비교·차트), 화면은 사람이 읽는다.

### 2.7 실패 처리

validate 실패 시 "저장 실패"가 아니라 **어떤 필드가 왜 걸렸는지 + 해당 필드만 재질문.** 예: "외부 이벤트는 출처가 1개 이상 필요합니다(V5). 기사 URL을 알려주세요." 전체 입력을 다시 시키는 순간 도구는 버려진다.

## 3. config.yaml 스키마 (신규 — I1)

```yaml
company:
  name: "메디테라피"
  official_urls: ["https://meditherapy.co.kr/"]
markets: [KR]                    # 확장 시 US, JP, SEA 추가
default_market: KR
channels: [own_mall, olive_young, sikor, duty_free, amazon, qoo10, shopee]
default_channels: [own_mall, olive_young]
competitors:                     # event-scan의 조사 범위를 정의
  - { name: "메디큐브(APR)", keywords: ["메디큐브", "AGE-R"] }
  - { name: "토리든", keywords: ["토리든"] }
  - { name: "라운드랩", keywords: ["라운드랩"] }
scan:
  lookback_weeks: 8              # simulate 기본 window와 공유
  extra_keywords: ["레티날", "PDRN", "홈에스테틱", "K뷰티 규제"]
  source_whitelist: []           # 비우면 등급제(tier)만 적용
```

`/setup`이 공식몰 URL과 `docs/research.md`를 읽어 초안을 만들고 사용자가 확인한다. 경쟁사 목록은 research.md §9의 경쟁 구도에서 초안 생성.

## 4. 스킬별 입력 플로우

### 4.1 `/setup` (신규 확정 — 6번째 스킬)

```
1. config.yaml 존재? → 있으면 "수정 모드"로 현재 값 표시
2. 공식몰 URL 또는 리서치 문서 경로 요청 (docs/research.md 자동 감지 시 제안)
3. 회사 프로필·경쟁사·채널 초안 생성 → 확인/수정
4. taxonomy.yaml 초안 생성 (공식몰 카테고리 기반) → 확인
5. 완료 메시지 + 다음 단계 안내: "/persona-build로 페르소나를 만드세요"
```

다른 모든 스킬은 시작 시 config 존재를 검사하고, 없으면 `/setup`을 안내하고 중단한다(빈 상태 처리, §6-2).

### 4.2 `/event-add`

```
1. (--external이면 외부 모드: 출처 URL 필수)
2. 자연어로 이벤트 설명 요청
3. 구조화 → 한국어 병기 확인 표 출력 → 수정 반영
4. 중복 검사(§2.5) → validate → 저장
5. 다음 단계 안내: "reviewed 이벤트가 N건입니다. /simulate로 반응을 추론하시겠어요?"
```

내부 이벤트는 날짜가 미래이거나 사용자가 "계획"이라고 말하면 `status: planned` 제안.

### 4.3 `/event-scan`

```
1. config의 competitors·keywords·lookback으로 조사 범위 구성 (플래그로 덮어쓰기 가능)
2. 웹 조사 → 후보 이벤트 구조화, 출처 tier 부여, 중복/기존 URL 스킵
3. unreviewed로 저장 후 검토 큐 테이블 출력:
   ┌─ # │ 유형 │ 제목 │ 날짜 │ 출처(tier) ─┐
4. 사용자가 번호로 승인/거부/보류 (예: "1,3 승인 2 거부")
5. 결과 요약 + 안내: "reviewed 5건. /simulate 또는 /brief 계속"
```

### 4.4 `/simulate`

- 기본: lookback window 내 `reviewed`(외부) + `occurred/planned`(내부) 이벤트 전체.
- `--event <id>`: 단일 이벤트만 — 긴급 대응(S3)·사전 점검(S2)용.
- `--backtest <id>`: 과거 이벤트 모드 — 해당 이벤트 시점 이후 데이터를 근거에서 차단하고 추론.
- 종료 시 **콘솔 미니 요약**(상위 리스크/기회 3개 + confidence)을 즉시 출력 — HTML을 열지 않아도 방향을 알 수 있어야 한다.

### 4.5 `/report`

- 기본: 최신 run + 직전 보고서와 diff. `--run <id>`, `--no-diff` 플래그.
- 산출: `reports/YYYY-MM-DD.html` 경로를 출력하고 브라우저로 열도록 안내.

### 4.6 `/brief` (신규 확정 — 7번째 스킬, 원커맨드 주간 루틴)

```
scan → [검토 큐에서 일시정지: 사용자 승인/거부] → simulate → report → 콘솔 요약 + HTML 경로
```

- 유일한 일시정지 지점은 검토 큐 — 승인 게이트(V6)는 자동화하지 않는다. 신뢰성 장치를 우회하는 자동화는 만들지 않는 것이 원칙.
- 신규 외부 이벤트가 0건이면 스캔 후 "변화 없음"을 보고하고 시뮬레이션을 건너뛸지 묻는다.

## 5. 사용 시나리오

### S0. 온보딩 (최초 1회, 목표: 30분 내 첫 보고서)

```
/setup (config·taxonomy, ~5분) → /persona-build (초안 8개 → 검수, ~15분)
→ /event-scan (첫 스캔 → 검토) → /simulate → /report
```
사용성 포인트: 각 단계가 끝날 때 다음 명령을 정확히 안내. 첫 보고서까지의 시간이 도구 채택을 결정한다.

### S1. 주간 브리핑 (핵심 반복 루틴)

```
매주 월요일: /brief → 검토 큐 승인 → (자동) → HTML 팀 공유
```
가치: 변화 추적 섹션(plan.md §6-4)이 "지난주 대비 무엇이 달라졌나"를 답한다. 사용자 조작은 검토 큐 응답 1회뿐.

### S2. 실행 전 사전 점검 (planned 내부 이벤트)

```
/event-add "다음 달 레티날 세럼 리뉴얼 출시 계획..." (status: planned)
→ /simulate --event <id> → 콘솔 요약에서 리스크 확인
→ 계획 수정 → --edit로 갱신 → 재시뮬레이션
→ 실행 후 /event-add --confirm <id> (occurred 전환)
```
가치: 도구가 사후 분석기가 아닌 의사결정 도구가 되는 시나리오. suggested_actions(M2)가 직접 소비되는 지점.

### S3. 긴급 대응 (외부 충격 즉시 분석)

```
경쟁사 대형 발표 인지 → /event-add --external (기사 URL과 함께)
→ /simulate --event <id> → 콘솔 미니 요약으로 즉시 방향 파악
→ 필요 시 /report로 공유용 문서화
```
사용성 포인트: 보고서 전체 사이클 없이 "이 이벤트 하나가 우리 페르소나에 무슨 의미인가"를 수 분 내 답한다.

### S4. 백테스트 (데모 및 신뢰 구축 — 데모는 이 시나리오 단독)

```
/simulate --backtest evt_2025_qoo10_no1 → /report
→ 추론 결과와 실제 이후 반응(보도·리뷰)을 나란히 확인
```
해커톤 데모 흐름: S0 축약(사전 준비된 데이터) → S4 → S1의 diff 화면.

### S5. 분기 페르소나 리프레시

```
/persona-build --refresh → 기존 대비 변경 diff 제안 → 항목별 승인
```
백본은 느리게 변한다(E3) — 자동 갱신하지 않고 항상 사람이 승인.

## 6. UX 원칙 (Codex 플러그인 공통)

1. **체이닝 안내**: 모든 스킬은 종료 시 상태 기반으로 다음 단계를 제안한다("unreviewed 3건 → 검토하시겠어요?").
2. **빈 상태 처리**: 전제조건(config→페르소나→이벤트→run) 누락 시 에러가 아니라 해당 단계로 유도.
3. **콘솔 요약 우선**: HTML은 공유·기록용, 즉답은 콘솔에서. 스킬이 "파일 생성됨"만 말하고 끝나면 안 된다.
4. **확인-후-저장**: LLM이 구조화한 모든 것은 저장 전 사용자 확인을 거친다(§2.1).
5. **승인 게이트 비자동화**: /brief를 포함해 어떤 자동화도 V6(검토 큐)을 건너뛰지 않는다.
6. **멱등 실행**: 같은 명령 재실행이 중복 데이터를 만들지 않는다(§2.5).
7. **실패는 필드 단위로**: 전체 재입력 요구 금지(§2.7).
