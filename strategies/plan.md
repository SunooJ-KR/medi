# LaunchPass — K-뷰티 글로벌 런칭 레디니스 검증 플러그인
## 메디테라피 해커톤 최종 기획서 (Codex Plugin)

---

## 0. 한 줄 정의

> **제품명과 타겟 국가를 입력하면, 현지 시장 조사 → 표현 규제 검증 → 현지 페르소나별 카피 생성 → 런칭 레디니스 판정까지 자동 수행하여 HTML 보고서를 생성하는 Codex 플러그인.**

**특정 국가를 기본값으로 내장하지 않는다.** 플러그인이 아는 국가는 없고, 실행 시 사용자가 입력하는 국가가 유일한 기준이다. 해당 국가의 concerns/rules/personas 데이터가 아직 없으면 그 자리에서 부트스트랩(웹서치 → 초안 생성 → 사람 승인)되어 즉시 사용 가능해진다. 코드(SKILL.md 절차, validate_copy.py)는 어떤 국가가 입력되어도 동일하게 동작하며, 국가마다 달라지는 것은 오직 디스크의 데이터 파일뿐이다.

## 1. 문제 정의 — 공개 자료 기반 3단 논증

### 논증 ① 메디테라피는 글로벌 확장이 핵심 전략이다 (사실)

- 2023년 7월 일본 진출, 2023년 4월 아마존(북미), 2023년 9월 쇼피(동남아) 런칭
- 글로벌 매출 비중이 빠르게 상승 중이며, 2024년 260억 시리즈A 투자의 핵심 명분도 글로벌 확장
- 2030년 누적매출 1조 목표 달성을 위해서는 해외 시장 성공이 필수
- 즉, 문제는 "일본 시장 하나"가 아니라 **진출하는 모든 해외 시장에서 반복적으로 발생**한다

### 논증 ② 해외 주요 화장품 시장은 광고 표현 규제가 존재한다 (사실)

- 화장품 광고에서 사용 가능한 효능 표현을 법으로 제한하는 체계는 국가마다 다른 이름으로 존재한다: 일본 약기법(薬機法), 미국 FDA 화장품 라벨링 규정 + FTC 광고 가이드, EU 화장품 규정(EC) No 1223/2009 등
- 예시로 일본 약기법은 화장품 효능 표현을 **56개 항목으로 명문화**하여 제한하며, "재생", "치료", "미백"(의약부외품 승인 없이는 불가), "주름이 사라진다" 등의 표현은 위반 시 **행정지도·과징금 대상**이다(2021년 개정으로 매출의 4.5% 과징금 도입). 위반 사례는 일본 후생노동성·소비자청이 공개 적발 사례로 발표한다(공개 자료로 입증 가능)
- 이는 일본에 국한된 특수 사례가 아니라, **"국가마다 다른 규칙으로 같은 종류의 검증이 반복 요구된다"는 구조적 패턴**의 한 예시일 뿐이다

### 논증 ③ 중소·중견 브랜드는 이를 시장마다 감당할 리소스가 없다 (구조적 문제)

- 대기업은 현지 법무팀·광고 심의 대행사를 시장별로 두지만, 메디테라피급 중견 브랜드는 제품 라인이 많고(링클핏, 비타, 에이클리어링, 슈마지, 블루레이어 등) 출시 주기가 짧아(2023년 한 해 3개 라인 출시) 진출하는 시장 수만큼 외부 검수 비용·속도 병목이 곱해진다
- 한국에서 쓰던 마케팅 카피("주름 개선", "미백 기능성", "피부 재생")를 그대로 번역하면, 그 표현이 허용되는지 여부는 국가마다 다르며 매번 새로 확인해야 한다
- 즉, **"한국식 표현을 타겟 국가의 규제에 맞게 검증·변환하는 작업"이 국가 불문 반복적·규칙기반·누락위험 높은 작업 → AI 자동화에 최적**. 이 구조 자체가 플러그인이 특정 국가에 종속되어선 안 되는 이유이기도 하다

## 2. 해결책 — LaunchPass 파이프라인

```
[입력]  제품명 + 제품 설명(성분/효능) + 타겟 국가 + 한국어 마케팅 카피(선택)

  │
  ▼
[STAGE 1 — 현지 시장 스캔]  (web_search)
  · 해당 국가 스킨케어 트렌드 · 유사 카테고리 경쟁 제품 · 가격대
  · 현지 소비자 피부 고민 우선순위
  │
  ▼
[STAGE 2 — 규제 검증 엔진]  (코드 실행 — 핵심 차별화)
  · 내장 룰셋(rules/{국가코드}.json)으로 카피 텍스트 검사
  · NG 표현 탐지 → 위반 유형 분류 → 대체 표현 제안
  · 검증 결과: PASS / WARNING / VIOLATION 판정
  │
  ▼
[STAGE 3 — 고민→페르소나→카피 파생]  (인과 체인)
  · 3a. concerns/{국가코드}.json 로드: 국가 × 나이대별 피부고민 우선순위
  · 3b. 제품 효능과 매칭되는 (나이대 × 고민) 셀 선정 → 타겟 페르소나 파생
  · 3c. 파생된 페르소나별 카피 생성 → STAGE 2 엔진에 재투입 (생성→검증 루프)
  │
  ▼
[STAGE 4 — 런칭 레디니스 리포트]  (HTML 생성)
  · 신호등 판정: 🟢 READY / 🟡 CONDITIONAL / 🔴 NOT READY
  · 시장 요약 + 규제 검증 결과표 + 페르소나별 승인 카피 + 액션 아이템
```

**데이터 흐름의 핵심 변화:** 페르소나를 미리 고정하지 않고, `concerns/{국가코드}.json`(국가 × 나이대 × 피부고민 격자)에서 **제품에 맞는 셀을 골라 페르소나를 파생**시킨다. 예를 들어 같은 제품이라도 A국에서는 "30대·모공" 페르소나로, B국에서는 "20대·자외선" 페르소나로 다르게 타겟팅될 수 있다 — 어느 국가가 A/B인지는 실행 시 입력에 따라 정해질 뿐, 코드에 정해져 있지 않다. 카피는 이 파생된 페르소나에 종속되어 생성된다.

### 핵심 설계 결정과 근거 (셀프 토론 결론)

| 쟁점 | 결정 | 근거 |
|------|------|------|
| 지원 범위 | 특정 국가 사전 내장 없음 — **입력 국가 기준 완전 모듈형** | 특정 국가를 기본값으로 삼으면 "왜 하필 이 나라인가" 질문에 답이 궁색해지고, 심사에서 그 국가에만 편향된 완성도로 오인될 수 있음. 스키마 + 부트스트랩 엔진만 제공하고, 시연 시 실제 입력으로 실시간 생성하는 편이 "살아있는" 데모가 됨 |
| 핵심 엔진 | 카피 생성이 아니라 **규제 검증** | 카피 생성만으론 "프롬프트 래퍼" 비판 회피 불가. 룰 기반 코드 검증은 Codex 코드 실행이 필수이고 결과가 검증 가능 |
| 페르소나 | 고정 나열 → **concerns 격자에서 동적 파생** | 국가마다 나이대별 고민이 달라, 제품에 맞는 셀에서 페르소나를 파생해야 타겟팅이 정확 |
| 카피 추천 | 페르소나의 copy_direction을 입력으로 생성 후 검증 루프 | 나이대·고민·규제를 모두 반영한 카피를 코드가 보증 |
| 판정 출력 | 신호등(적/황/녹) 레디니스 판정 | 심사 시연에서 "무엇이 해결됐는지" 즉시 전달됨 |
| 검증 방법 | 생성 카피를 검증 엔진에 재투입하는 자가 루프 | AI 출력의 품질을 AI가 아닌 **코드(룰셋)** 가 보증하는 구조 |

### 4-2. `rules/{국가코드}.json` — 부트스트랩으로 생성되는 룰셋 (예시: 일본 입력 시 산출물)

이 파일은 미리 만들어 저장소에 포함되는 파일이 아니라, 사용자가 **실행 시 "일본" 같은 국가를 입력했을 때** STAGE 0의 부트스트랩 절차가 생성하는 산출물이다. 아래는 그 산출물의 형태를 보여주는 예시일 뿐, 코드에 내장된 기본값이 아니다. 국가별 규제 공개 자료가 룰셋으로 구조화되므로, **생성된 파일 자체가 "공개 자료 기반" 증거물**이 된다.

### 4-3. `validate_copy.py` — 규제 검증 엔진

Codex가 직접 실행하는 파이썬 스크립트. **AI의 판단이 아닌 코드의 판정**이라는 점이 신뢰성의 근거.

```
기능:
1. 입력: 카피 텍스트(한국어/일본어) + 룰셋 JSON
2. 처리:
   - 패턴 매칭 (정규식 + 형태소 단위)
   - 한국어 카피의 경우: NG 개념 매핑 테이블로 사전 탐지
     ("재생" → JP-001 해당 예정 경고)
3. 출력(JSON):
   {
     "verdict": "VIOLATION",
     "violations": [
       {"rule_id": "JP-001", "matched": "피부 재생", "position": 12,
        "severity": "VIOLATION", "alternatives": [...]}
     ],
     "pass_rate": "3/5 문장 통과"
   }
```

### 4-4. `concerns/{국가코드}.json` — 국가 × 나이대 × 피부고민 격자 (예시: 일본 입력 시 산출물)

페르소나의 **상위 데이터**. 이 파일도 실행 시 입력된 국가에 대해 부트스트랩으로 생성되며, 일본은 그 형태를 보여주는 예시일 뿐이다. 국가마다 피부 고민 우선순위가 다르고, 같은 국가 안에서도 나이대별로 다르다는 사실을 격자로 구조화한다. 페르소나는 여기서 파생되므로, 이 파일이 "왜 이 페르소나인가"의 근거를 보유한다. 모든 셀에 공개 출처(evidence)를 명시해 "지어낸 데이터" 비판을 차단한다.

```json
{
  "market": "JP",
  "version": "1.0.0",
  "last_verified": "2026-07-05",
  "age_bands": ["10-20", "20-30", "30-40", "40-50", "50+"],
  "grid": [
    {
      "age_band": "20-30",
```

### 4-5. `personas/{국가코드}.json` — 국가별 페르소나 (예시: 일본 입력 시 concerns에서 파생된 산출물)

### 4-6. `SKILL.md` — 실행 절차서

```
# LaunchPass Skill

## 트리거
"{국가} 런칭 검증", "{국가} 진출 분석", "카피 규제 검사", "LaunchPass 실행"
(국가 미지정 시 반드시 먼저 확인: "어느 국가를 검증할까요?")

## STAGE 0 — 국가 리졸브
1. 사용자 입력에서 국가명을 국가코드로 변환 (일본→jp, 미국→us, EU→eu ...)
2. concerns/{국가코드}.json, rules/{국가코드}.json, personas/{국가코드}.json 존재 여부 확인
3. 하나라도 없으면 → 부트스트랩 절차(4-8 A)로 분기 후 이후 스테이지 진행

## STAGE 1 — 시장 스캔
web_search로 다음 조사 (국가코드에 따라 쿼리의 지역·언어만 치환, 절차는 고정):
- "{제품 카테고리} {국가} 트렌드 {현재년월}"
- "{제품 카테고리} {국가} 경쟁 제품 가격"
- 결과를 market_scan 섹션용으로 요약

## STAGE 2 — 규제 검증 (필수 코드 실행)
1. 사용자의 한국어 카피가 있으면 그대로, 없으면 제품 설명에서 예상 카피 추출
2. `python scripts/validate_copy.py --input {카피} --rules rules/{국가코드}.json` 실행
   (스크립트가 last_verified 90일 초과 시 경고 플래그 반환 → 보고서에 표시)
3. 결과 JSON을 파싱하여 위반 항목 정리
⚠️ 이 단계는 반드시 스크립트를 실행할 것. AI 자체 판단으로 대체 금지.
⚠️ 룰셋 본문을 컨텍스트로 읽지 말 것. 검증은 스크립트에 위임하고 결과만 사용.

## STAGE 3 — 고민→페르소나→카피 파생 + 자가 검증 루프
1. concerns/{국가코드}.json 로드 (없으면 부트스트랩 분기)
2. 제품 효능/성분과 격자의 top_concerns를 매칭 → 관련 (나이대 × 고민) 셀 선정
3. 선정된 셀에서 페르소나 파생 (personas/{국가코드}.json의 파생 규칙 적용).
   페르소나 개수를 코드에 가정하지 말 것 — 매칭된 셀 수만큼 동적으로 생성
4. 각 페르소나의 copy_direction을 입력으로 현지어 카피 생성
   (STAGE 2에서 확인된 NG 표현 회피, 해당 나이대·고민에 소구)
5. 생성된 카피 전건을 validate_copy.py --rules rules/{국가코드}.json 에 재투입
6. VIOLATION 발생 시 해당 카피 재생성 (최대 3회 루프)
7. 전 카피 PASS 시에만 다음 단계 진행

## STAGE 4 — 레디니스 리포트 생성
templates/readiness_report.html 기반으로 생성:
- 판정 로직: VIOLATION 미해결 → 🔴 / WARNING만 존재 → 🟡 / 전체 PASS → 🟢
- 저장: launchpass_report_{제품명}_{날짜}.html
```

### 4-8. 데이터 라이프사이클 — 국가 확장과 규제/트렌드 변경 대응

concerns·rules·personas 세 데이터셋 모두 **플러그인이 관리하는 자산**이다. 두 개의 보조 절차를 SKILL.md에 추가한다.

#### (A) 부트스트랩 — 새 국가 추가 (concerns + rules + personas)

```
트리거: "미국 시장 추가", "EU 룰셋 만들어줘"

1. [고민 격자] web_search로 해당 국가 나이대별 스킨케어 고민 공개 조사 리포트 수집
   → concerns/_schema.json에 맞춰 concerns/{국가코드}.draft.json 생성
   각 셀에 evidence(출처) 필수. 이 파일이 페르소나 파생의 원천이므로 가장 먼저 만든다

2. [규제] web_search로 해당 국가 화장품 광고 규제 공식 문서 조사
   (예: 미국 FDA 화장품 라벨링 + FTC 광고 가이드)
   → rules/_schema.json에 맞춰 rules/{국가코드}.draft.json 생성
   각 룰에 반드시 출처 URL 명시. 출처를 못 찾은 룰은 "unverified" 플래그

3. [페르소나] 1번 concerns 격자를 기반으로 나이대×고민 셀에서 페르소나 초안 파생
   → personas/_schema.json에 맞춰 personas/{국가코드}.draft.json 생성
   각 페르소나에 linked_concerns·age_band·evidence 필수, 근거 없으면 생성하지 않음

4. ⚠️ 사람 검토 요청: 세 draft를 사용자에게 제시하고 승인받은 후에만
   .draft 제거 → 정식 파일로 승격 (verified_by: "human" 기록)
```

세 파일 모두 같은 원칙을 따른다: **AI는 초안만 만들고, 확정은 사람이 한다.** 컴플라이언스·타겟팅 기준을 AI가 단독 확정하는 것은 법적·전략적 책임 관점에서 위험하므로, 이는 의도된 설계다. 해커톤 Q&A 방어 논리로도 사용한다. 생성 순서(concerns → rules → personas)는 의존성을 반영한다.

#### (B) 신선도 체크 — 규제·트렌드 변경 대응

```
1. 매 실행 시 validate_copy.py가 각 데이터셋의 last_verified 확인
   → 90일 초과 시 보고서 헤더에 "⚠️ 데이터 기준일 경과" 경고 표시
2. 사용자가 "업데이트" 요청 시 (rules/concerns 공통):
   a. source_urls의 공식 문서를 web_fetch로 재조회
   b. 현행 파일과 비교하여 변경점 diff만 생성 (신설/개정/폐지)
   c. diff를 사람에게 제시 → 승인 시 버전 증가(1.0.0 → 1.1.0) + last_verified 갱신
   d. concerns가 갱신되면 파생된 personas도 재검토 대상으로 표시
3. 모든 보고서에 "데이터 버전 · 기준일" 명시 → 판정의 시효를 투명화
```

#### 토큰 효율 설계

이 구조가 토큰 관점에서 유리한 이유:

| 설계 | 토큰 효과 |
|------|----------|
| 검증은 코드(validate_copy.py)가 디스크의 JSON을 직접 읽음 | 룰이 30개든 300개든, 국가가 10개든 **룰셋 본문이 LLM 컨텍스트에 진입하지 않음** |
| 검증 결과는 위반 항목만 반환 | PASS된 룰은 토큰 소비 0 |
| concerns 격자에서 제품 관련 셀만 선택 | 전체 격자가 아니라 매칭 셀만 컨텍스트로 |
| 국가별 파일 분리 (jp/us/sg...) | 실행 시 대상 국가 3개 파일만 로드, 나머지는 디스크에 존재만 |
| 업데이트는 diff만 컨텍스트에 올림 | 전체 재생성 대비 대폭 절감 |
| 부트스트랩은 국가당 1회성 비용 | 이후 실행에서는 재조사 없이 로컬 데이터 재사용 |

## 5. 질문지 5문항 답변 초안

**Q1. 무엇을, 누가, 어떤 상황에서 쓰나요?**
K-뷰티 브랜드(메디테라피)의 마케터·해외사업 담당자가, 제품을 특정 해외 시장(실행 시 입력하는 국가)에 출시하기 전 마케팅 카피의 규제 적합성을 검증하고 현지 페르소나별 승인 카피를 확보해야 하는 상황에서 사용합니다.

**Q2. 왜 이 문제를 선택했나요?**
메디테라피는 일본·북미·동남아 등 복수 해외 시장에 진출한 상태이며 글로벌 매출 비중이 상승 중입니다(공개 자료). 화장품 광고 표현 규제는 국가마다 다른 이름과 기준으로 존재하며(예: 일본 약기법은 위반 시 매출의 4.5% 과징금, 적발 사례도 공개), 진출 시장이 늘수록 이 검증 부담은 국가 수만큼 곱해집니다. 제품 라인이 많고 출시 주기가 짧은 중견 브랜드가 시장마다 매번 외부 법무 검수를 거치는 것은 비용·속도 병목입니다. 반복적이고 규칙 기반이며 국가 불문 누락 위험이 큰 작업 — AI 에이전트 자동화에 최적인 문제이자, 특정 국가에 종속되어선 안 되는 이유이기도 합니다.

**Q3. 플러그인은 어떻게 작동하나요?**
5단계 파이프라인: ① 국가 리졸브 — 입력 국가에 대응하는 concerns/rules/personas 데이터가 없으면 그 자리에서 부트스트랩(웹서치 → 초안 → 사람 승인) → ② web_search로 현지 시장 스캔 → ③ 파이썬 검증 엔진(validate_copy.py)이 국가별 룰셋(rules/{국가코드}.json) 기반으로 카피의 NG 표현을 코드로 탐지 → ④ concerns 격자에서 제품에 맞는 (나이대×고민) 셀을 골라 페르소나를 파생하고 카피를 생성해 검증 엔진에 재투입하는 자가 검증 루프 → ⑤ 신호등 판정이 포함된 HTML 레디니스 보고서 생성. 어떤 국가가 입력되어도 동일한 절차가 적용되며, 저장소에는 특정 국가의 데이터가 미리 포함되어 있지 않습니다.

**Q4. AI를 어떻게 활용했나요?**
AI는 시장 조사(검색·요약), 카피 생성(페르소나 맞춤), 위반 표현의 대체안 제안을 담당합니다. 단, 규제 위반 판정 자체는 AI 판단이 아니라 공개 법령 기반 룰셋 + 코드 실행으로 수행하여, AI 환각이 컴플라이언스 판정에 개입하지 못하게 설계했습니다. 생성(AI) → 검증(코드) → 재생성(AI) 루프가 핵심 AI 활용 구조입니다.

**Q5. 어떻게 검증했나요?**
① 룰셋 단위 테스트: 알려진 위반 표현 샘플("피부 재생", "美白" 등)이 정확히 탐지되는지 스크립트 테스트. ② 실제 사례 검증: 메디테라피 링클핏 라인의 실제 한국어 제품 설명을 입력하고 타겟 국가를 임의로 지정(테스트 케이스로 일본 선택)해 위반 탐지 → 수정 → PASS 전환 과정을 로그로 기록. 이 과정에서 국가 데이터는 저장소에 미리 있던 게 아니라 부트스트랩으로 그 실행 중에 생성된 것임을 로그로 확인. ③ 자가 루프 검증: AI가 생성한 카피가 검증 엔진을 100% 통과할 때까지 재생성되는지 확인.

## 6. 원래 기획 대비 개선 포인트 정리

| 항목 | 기존 (BeautySignal) | 개선 (LaunchPass) |
|------|--------------------|--------------------|

| 시연 스토리 | "이렇게 반응할 것 같습니다" | "이 카피는 이 조항 위반입니다 → 수정 → 통과" |
