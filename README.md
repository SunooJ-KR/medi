# 고객반응 시뮬레이터 (Persona × Event Reaction Simulator)

> 회사 안팎에서 벌어지는 이벤트가 **피부고민별 고객 페르소나**의 의사결정에 어떤 영향을 주는지 구조적으로 추론하고, 웹 보고서(HTML)로 정리해 주는 Codex 플러그인입니다.
> 사례 대상: 뷰티 D2C 브랜드 **메디테라피(MEDITHERAPY)** — 근거는 공개 자료(`docs/research.md`)로 검증했습니다.

---

## 이 도구가 푸는 문제

뷰티 시장은 제품과 트렌드가 빠르게 바뀌지만, 고객이 겪는 피부 고민(민감·모공·잡티·주름·건조·트러블)은 잘 변하지 않습니다. 반대로 시장 이벤트 — 자사 프로모션, 경쟁사 출시, 플랫폼 정책, 규제, 인플루언서 이슈 — 는 매주 쏟아집니다. 마케터 한 명이 "이 이벤트가 우리 고객층 각각에게 무슨 의미인가"를 매번 빠짐없이 따지기는 어렵습니다.

이 플러그인은 그 판단을 돕습니다.

- **불변 백본**: 피부고민 택소노미 + 근거로 뒷받침된 고객 페르소나 6~8개 (느리게 변함)
- **동적 오버레이**: 내부 이벤트(직접 입력) + 외부 이벤트(자동 조사) (매주 변함)
- **출력**: 이벤트 × 페르소나 쌍마다, 행동과학 프레임워크(Fogg B=MAP: 동기·능력·계기)에 따라 반응이 어떻게 바뀌는지 + **그래서 무엇을 할지**(추천 액션)

> ⚠️ **중요한 성격**: 이 도구는 "미래 예측기"가 아니라 **근거가 달린 가설 생성기**입니다. 모든 반응에는 출처 인용과 확신도(confidence)가 붙고, 확신이 낮은 항목은 보고서에서 흐리게 표시됩니다. 과거 실제 이벤트로 추론을 검증하는 **백테스트** 기능으로 신뢰도를 확인할 수 있습니다.

## 요구 사항

| 항목 | 버전/조건 |
|---|---|
| Codex CLI | 설치·로그인 완료 (`codex login`) |
| Node.js | v18 이상 (로컬 v24 확인됨) |
| OS | Windows / macOS / Linux |

플러그인 내부 의존성은 `js-yaml`, `ajv` 두 개뿐이며, 설치 시 자동으로 잡힙니다. 보고서는 외부 CDN·네트워크 없이 열리는 단일 HTML 파일로 생성됩니다.

## 설치

```bash
# 이 저장소를 Codex 플러그인으로 추가
codex plugin install ./src

# 스크립트 의존성 설치
cd src && npm install && cd ..
```

설치가 끝나면 Codex 세션에서 아래 스킬(슬래시 명령)을 쓸 수 있습니다.

## 빠른 시작 — 첫 보고서까지

처음 한 번은 온보딩이 필요합니다. 목표는 **30분 내 첫 보고서**입니다.

```text
/setup            # ① 회사 프로필·경쟁사·채널·피부고민 택소노미 초안 생성 → 확인
/persona-build    # ② 공개 자료에서 고객 페르소나 6~8개 초안 → 항목별 검수·승인
/event-scan       # ③ 외부 이벤트 자동 조사 → 검토 큐에서 승인/거부
/simulate         # ④ 반응 추론 → 콘솔 요약 즉시 출력
/report           # ⑤ HTML 보고서 생성 → 브라우저로 열기
```

각 스킬은 끝날 때 **다음에 실행할 명령을 안내**하므로 순서를 외울 필요가 없습니다. 설정이 안 된 상태에서 다른 스킬을 부르면 `/setup`으로 안내하고 멈춥니다.

## 명령어(스킬) 레퍼런스

| 명령 | 하는 일 | 자주 쓰는 옵션 |
|---|---|---|
| `/setup` | 회사 프로필(`config.yaml`)·피부고민 택소노미 초안 생성 및 확인 | — |
| `/persona-build` | 공개 자료·리뷰에서 페르소나 초안 생성, 사람이 검수해 확정 | `--refresh`(분기 갱신) |
| `/event-add` | 이벤트를 자연어로 입력 → 구조화 확인 → 저장 | `--external`(수동 외부, 출처 URL 필수), `--edit <id>`, `--confirm <id>`(계획→실행 전환) |
| `/event-scan` | 경쟁사·키워드·기간 기준으로 외부 이벤트 자동 조사 → 검토 큐 | — |
| `/simulate` | 이벤트 × 페르소나 반응 추론 후 저장, 콘솔 요약 출력 | `--event <id>`(단일), `--backtest <id>`(과거 검증), `--since <날짜>` |
| `/report` | 최신 결과로 HTML 보고서 생성 + 직전 보고서와 변화 비교 | `--run <id>`, `--no-diff` |
| `/brief` | 주간 루틴을 한 번에: 조사→(검토 승인)→추론→보고서 | — |

## 이렇게 씁니다 — 주요 시나리오

**주간 브리핑 (핵심 루틴).** 매주 한 번 `/brief`를 실행하면 외부 조사부터 보고서까지 자동으로 이어집니다. 중간에 딱 한 번, **검토 큐에서 어떤 외부 이벤트를 반영할지 승인**하는 지점에서만 멈춥니다(신뢰성을 위해 이 승인은 자동화하지 않습니다). 보고서의 "변화 추적" 섹션이 지난주 대비 무엇이 달라졌는지 알려 줍니다.

**실행 전 사전 점검.** 프로모션이나 신제품 출시를 **실행하기 전에** 계획 상태(`planned`)로 입력하고 `/simulate --event <id>`를 돌리면, 각 페르소나가 어떻게 반응할지 미리 볼 수 있습니다. 계획을 수정해 다시 돌려 보고, 실제 실행 후 `/event-add --confirm`으로 확정합니다.

**긴급 대응.** 경쟁사 대형 발표를 인지했을 때 `/event-add --external`(기사 URL과 함께)로 넣고 `/simulate --event <id>`를 돌리면, 전체 보고서 없이도 수 분 내에 "이 사건이 우리 페르소나에 무슨 의미인가"를 콘솔 요약으로 확인합니다.

**백테스트 (신뢰도 검증).** `/simulate --backtest <과거이벤트>`는 그 이벤트 발생 시점 이후의 정보를 근거에서 차단한 채 추론합니다. 결과를 실제로 알려진 이후 반응(리뷰·보도)과 나란히 비교해, 추론 절차가 믿을 만한지 확인할 수 있습니다.

## 결과 확인 방법

### 1) 콘솔 요약 (즉답)

`/simulate`와 `/brief`는 실행이 끝나면 **상위 리스크/기회 3개와 확신도**를 콘솔에 바로 출력합니다. HTML을 열지 않아도 방향을 알 수 있습니다.

### 2) HTML 보고서 (공유·기록용)

`/report`는 `reports/YYYY-MM-DD.html`에 단일 HTML 파일을 만듭니다. 파일 하나만 복사하면 다른 PC·팀 채널에서도 그대로 열립니다. 5개 섹션으로 구성됩니다.

| 섹션 | 내용 |
|---|---|
| **요약** | 이번 회차 핵심 3줄 + **대응 우선순위 Top 3** + 신규/승인대기 이벤트 수 |
| **시각화** | 페르소나 × 이벤트 히트맵, 이벤트 타임라인, 피부고민별 기회–리스크 매트릭스 |
| **상세** | 페르소나 카드(B=MAP + 근거 인용), 이벤트 카드(출처 링크·등급), 반응 상세(추천 액션 포함) |
| **변화 추적** | 직전 보고서 대비 신규 이벤트·반응 변화·우선순위 변동 |
| **한계 고지** | 합성 페르소나의 한계, 미검토 이벤트 수 등 신뢰도 유의사항 |

읽을 때 참고할 점:
- **모든 주장 옆에 근거 링크**가 있습니다. 출처 없는 판단은 없습니다.
- **확신도가 낮은(low) 반응은 흐리게** 표시됩니다 — 과신하지 않도록 하는 장치입니다.
- **계획(planned) 이벤트는 "사전 점검" 배지**로 구분되며, 변화 추적·백테스트 통계에서는 제외됩니다.

### 3) 원본 데이터

모든 페르소나·이벤트·반응은 작업 폴더의 `data/`에 사람이 읽는 YAML로 저장되어 git으로 이력이 남습니다. 보고서 숫자의 근거를 직접 확인하거나 수정할 수 있습니다.

```text
data/
  config.yaml          # 회사 프로필·경쟁사·채널·조사 범위
  taxonomy.yaml        # 피부고민 택소노미(성분·대표 제품 매핑)
  personas/            # 고객 페르소나 (백본)
  events/internal/     # 내부 이벤트 (직접 입력)
  events/external/     # 외부 이벤트 (자동/수동)
  reactions/<run-id>/  # 시뮬레이션 결과
reports/               # 날짜별 HTML 보고서
```

## 신뢰성은 어떻게 지키나

이 도구는 그럴듯한 서사를 만들어내기 쉬운 LLM의 약점을 **코드 검증(validate)으로** 막습니다.

- **근거 없는 데이터는 저장 거부**: 근거 인용이 없는 페르소나, 출처가 없는 외부 이벤트는 파일로 저장되지 않습니다.
- **사람 승인 게이트**: 자동 수집한 외부 이벤트는 검토 큐에서 승인해야만 시뮬레이션에 반영됩니다. `/brief`도 이 지점만은 자동화하지 않습니다.
- **구조화된 반응**: 반응은 자유 서술이 아니라 -2~+2 정수 델타(동기·능력·계기)로 표현되어, 회차 간 비교와 시각화가 가능합니다.
- **확신도 3단계 + 흐림 렌더**: 근거의 직접성에 따라 high/medium/low를 매기고, 낮은 항목은 UI에서 눈에 덜 띄게 처리합니다.
